# üß™ What is a Pipe in NestJS?

**Pipes** are used for:

- ‚úÖ Data transformation (e.g., convert `string` to `number`)
- ‚úÖ Data validation (e.g., check if value is not empty)

They are executed **after the controller receives the request arguments**, but **before the method is invoked**.

---

# üîÑ NestJS Lifecycle Summary

Understanding when pipes run:

1. **Request hits route**
2. **Guards run first** (can block request)
3. **Interceptors run**
4. **Pipes run next** üî•
5. **Controller method executes**
6. **Response passes through interceptors (again)**

---

# üîß Where Can You Use Pipes?

| Scope      | Usage                                  |
| ---------- | -------------------------------------- |
| Parameter  | `@Param('id', ParseIntPipe)`           |
| Route      | `@UsePipes(new ValidationPipe())`      |
| Controller | `@UsePipes(...)`                       |
| Global     | `app.useGlobalPipes(...)` in `main.ts` |

# üîå Built-in Pipes in NestJS

---

## 1. **ParseIntPipe**

üîπ Transforms a string to an integer.

```ts
@Get(':id')
getUser(@Param('id', ParseIntPipe) id: number) {
  return `User ID is ${id}`;
}
```

üìõ **Common Mistake:**

- Sending `/user/abc` ‚Üí will throw `BadRequestException`.

---

## 2. **ParseBoolPipe**

üîπ Transforms a string (`true`, `false`, `1`, `0`) to boolean.

```ts
@Get()
getActiveUsers(@Query('active', ParseBoolPipe) active: boolean) {
  return active ? 'Active users' : 'All users';
}
```

üìõ **Mistake:** Boolean strings like `"yes"` or `"no"` will fail.

---

## 3. **ParseArrayPipe**

üîπ Converts a comma-separated string to an array.

```ts
@Get()
filterUsers(@Query('roles', new ParseArrayPipe({ items: String })) roles: string[]) {
  return roles; // ["admin", "user"]
}
// ?roles=admin,user
```

‚úÖ You can also validate the array length, type, delimiter, etc.

---

## 4. **ParseUUIDPipe**

üîπ Validates and transforms a UUID string.

```ts
@Get(':uuid')
findByUUID(@Param('uuid', ParseUUIDPipe) uuid: string) {
  return uuid;
}
```

---

## 5. **DefaultValuePipe**

üîπ Assigns a default value if the input is `undefined` or empty.

```ts
@Get()
getPaginated(
  @Query('page', new DefaultValuePipe(1), ParseIntPipe) page: number,
) {
  return page; // 1 if not provided
}
```

‚úÖ Combine with other pipes!

---

## 6. **ValidationPipe**

üîπ Validates request bodies against **class-validator** decorated DTOs.

### DTO:

```ts
import { IsString, IsNotEmpty } from "class-validator";

export class CreateUserDto {
  @IsString()
  @IsNotEmpty()
  name: string;
}
```

### Controller:

```ts
@Post()
@UsePipes(new ValidationPipe({ whitelist: true }))
createUser(@Body() dto: CreateUserDto) {
  return dto;
}
```

‚úÖ `whitelist: true` strips out unrecognized fields
‚úÖ `forbidNonWhitelisted: true` will throw an error if extra fields exist

üìõ **Mistake**: Not importing `ValidationPipe` or forgetting to use `@Is...()` decorators

---

# ‚öôÔ∏è Create a Custom Pipe

```ts
import { PipeTransform, Injectable, BadRequestException } from "@nestjs/common";

@Injectable()
export class UppercasePipe implements PipeTransform {
  transform(value: any) {
    if (typeof value !== "string") {
      throw new BadRequestException("Expected string");
    }
    return value.toUpperCase();
  }
}
```

### Usage:

```ts
@Get(':name')
getUpper(@Param('name', UppercasePipe) name: string) {
  return name; // returns name in uppercase
}
```

---

# üß† Best Practices

‚úÖ Use **DTOs** with `ValidationPipe` for any `@Body()` request

‚úÖ Chain pipes together:

```ts
@Query('page', new DefaultValuePipe(1), ParseIntPipe)
```

‚úÖ Always enable global validation in `main.ts`:

```ts
app.useGlobalPipes(new ValidationPipe({ whitelist: true }));
```

‚úÖ Write **custom pipes** for reusable data transformation or validation logic

# ‚ùå Common Mistakes

| Mistake                           | Why it's wrong                             |
| --------------------------------- | ------------------------------------------ |
| Using `ParseIntPipe` on `@Body()` | It‚Äôs meant for query/param strings         |
| Forgetting `@Is...()` decorators  | DTO validation won‚Äôt run                   |
| Skipping `ValidationPipe`         | NestJS won‚Äôt validate your input           |
| Not chaining pipes                | You may forget defaults or transformations |

## ‚úÖ 1. Pipes for Transforming Nested Objects

### üîç Problem:

By default, DTO validation works for top-level fields. But nested objects/arrays need explicit handling.

---

### ‚úÖ DTO with Nested Validation

```ts
// address.dto.ts
import { IsString } from "class-validator";

export class AddressDto {
  @IsString()
  city: string;

  @IsString()
  country: string;
}
```

```ts
// user.dto.ts
import { Type } from "class-transformer";
import { IsString, ValidateNested } from "class-validator";
import { AddressDto } from "./address.dto";

export class CreateUserDto {
  @IsString()
  name: string;

  @ValidateNested()
  @Type(() => AddressDto) // <-- IMPORTANT
  address: AddressDto;
}
```

---

### ‚úÖ Controller

```ts
@Post()
createUser(@Body() body: CreateUserDto) {
  return body;
}
```

### üõ† Requirements:

- `@ValidateNested()` ‚Üí Enables validation inside the object
- `@Type(() => Class)` from `class-transformer` ‚Üí Tells Nest how to construct nested types

---

## ü™Ñ Bonus: Nested Arrays

```ts
@ValidateNested({ each: true })
@Type(() => AddressDto)
addresses: AddressDto[];
```

## ‚úÖ 2. Global Pipes Based on Request Type

### üß† Goal:

Apply **different pipes** depending on the **route**, **method**, or **headers**.

### üõ† Solution:

Use a **custom global pipe** that checks request metadata:

```ts
import {
  Injectable,
  PipeTransform,
  ArgumentMetadata,
  ValidationPipe,
} from "@nestjs/common";

@Injectable()
export class ConditionalPipe implements PipeTransform {
  private validationPipe = new ValidationPipe({
    whitelist: true,
  });

  async transform(value: any, metadata: ArgumentMetadata) {
    if (
      metadata.type === "body" &&
      metadata.metatype?.name === "CreateUserDto"
    ) {
      return this.validationPipe.transform(value, metadata);
    }

    return value;
  }
}
```

### Usage in `main.ts`:

```ts
app.useGlobalPipes(new ConditionalPipe());
```

üìå This way, you avoid applying heavy validation pipes to all routes.

## ‚úÖ 3. Testing Custom Pipes

### üß™ Use Case:

You wrote a pipe like `TrimPipe`, `UppercasePipe`, or `PositiveNumberPipe`, and want unit tests.

---

### Example Pipe

```ts
import { PipeTransform, Injectable, BadRequestException } from "@nestjs/common";

@Injectable()
export class PositiveNumberPipe implements PipeTransform {
  transform(value: any) {
    const num = Number(value);
    if (isNaN(num) || num <= 0) {
      throw new BadRequestException("Must be a positive number");
    }
    return num;
  }
}
```

---

### üß™ Test File (Jest)

```ts
import { PositiveNumberPipe } from "./positive-number.pipe";
import { BadRequestException } from "@nestjs/common";

describe("PositiveNumberPipe", () => {
  const pipe = new PositiveNumberPipe();

  it("should transform valid number", () => {
    expect(pipe.transform("5")).toBe(5);
  });

  it("should throw on zero", () => {
    expect(() => pipe.transform("0")).toThrow(BadRequestException);
  });

  it("should throw on negative number", () => {
    expect(() => pipe.transform("-1")).toThrow(BadRequestException);
  });

  it("should throw on non-numeric string", () => {
    expect(() => pipe.transform("abc")).toThrow(BadRequestException);
  });
});
```

## ‚úÖ 1. Pipes with Request Context (e.g. user ID, language, IP)

Normally, pipes receive just the `value` and `metadata`, **not the full request object**.

But there's a workaround: use `ExecutionContext` via a **custom decorator and `ArgumentMetadata.data`**.

---

### ‚úÖ Use Case:

- Validate `@Body()` based on the authenticated user
- Validate or transform input based on `Accept-Language`, `x-user-id`, etc.

---

### Step-by-step:

#### 1. Create a **Custom Pipe**

```ts
import {
  PipeTransform,
  Injectable,
  ArgumentMetadata,
  BadRequestException,
  Inject,
} from "@nestjs/common";
import { Request } from "express";

@Injectable()
export class LanguageAwarePipe implements PipeTransform {
  transform(value: any, metadata: ArgumentMetadata) {
    const request: Request = metadata.data;

    const lang = request.headers["accept-language"] || "en";

    if (lang === "fa") {
      // Do something like translating or validating Farsi-specific data
    }

    return value;
  }
}
```

---

#### 2. Use it in Controller

```ts
@Post()
createSomething(
  @Body(LanguageAwarePipe) body: any,
  @Req() req: Request,
) {
  // pipe will receive the same `req` via ArgumentMetadata.data
}
```

‚ö†Ô∏è But by default, `@Body()` only passes the body value. To pass the full `req`, we need this pattern:

---

### Alternative (cleaner): Use Interceptors + Context

You can create a custom pipe that accepts a request via constructor:

```ts
@Injectable()
export class UserContextPipe implements PipeTransform {
  constructor(private readonly req: Request) {}

  transform(value: any) {
    const userId = this.req.headers["x-user-id"];
    if (!userId) {
      throw new BadRequestException("Missing user id");
    }

    value.userId = userId;
    return value;
  }
}
```

But this **only works with DI** if you use it globally or in providers. If used inline in a controller method, you‚Äôll need a factory:

```ts
@UsePipes(new UserContextPipe(req)) // unsafe unless wrapped in a provider
```

## ‚úÖ 2. Use Zod or Yup for Schema Validation in NestJS Pipes

### üí° Why?

- More powerful than `class-validator`
- Works with `schema.safeParse(data)`
- Replaces the need for DTO classes

---

### ‚úÖ Example with Zod

#### 1. Install Zod:

```bash
npm install zod
```

---

#### 2. Create a ZodValidationPipe

```ts
import { PipeTransform, Injectable, BadRequestException } from "@nestjs/common";
import { ZodSchema } from "zod";

@Injectable()
export class ZodValidationPipe implements PipeTransform {
  constructor(private schema: ZodSchema) {}

  transform(value: any) {
    const result = this.schema.safeParse(value);
    if (!result.success) {
      throw new BadRequestException(result.error.format());
    }
    return result.data;
  }
}
```

---

#### 3. Define a Schema and Use it

```ts
import { z } from "zod";

const createUserSchema = z.object({
  name: z.string().min(1),
  email: z.string().email(),
  age: z.number().optional(),
});
```

```ts
@Post()
createUser(
  @Body(new ZodValidationPipe(createUserSchema)) body: any,
) {
  return body;
}
```

‚úÖ You now validate request bodies without creating a DTO class.

---

### ‚úÖ Example with Yup

You can do the same with **Yup**:

```bash
npm install yup
```

Create `YupValidationPipe`:

```ts
import { PipeTransform, Injectable, BadRequestException } from "@nestjs/common";
import * as yup from "yup";

@Injectable()
export class YupValidationPipe implements PipeTransform {
  constructor(private schema: yup.ObjectSchema<any>) {}

  async transform(value: any) {
    try {
      const validated = await this.schema.validate(value, {
        abortEarly: false,
      });
      return validated;
    } catch (err: any) {
      throw new BadRequestException(err.errors);
    }
  }
}
```
